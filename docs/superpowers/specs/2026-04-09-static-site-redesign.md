# VB Analyzer — Static Site Redesign

## Goal

Convert the vb-analyzer from a Dash server app to a polished static site deployable on GitHub Pages. Improve the frontend design to match the professional look of the NCVA Power League Dashboard — dark theme, micro-interactions, polished typography, and overall visual refinement. Use the frontend-design skill for creative execution.

## Architecture

**Approach: Python generates JSON, separate HTML/CSS/JS frontend.**

The existing analytics Python modules stay untouched. A new `generate_site.py` script calls them and exports pre-computed JSON. A separate `site/` folder contains the static frontend that loads these JSON files and renders charts with Plotly.js.

```
vb-analyzer/
├── analytics/           # existing, unchanged
│   ├── core.py          # rally/action DataFrames, score classification
│   ├── player.py        # player stats, clutch, consistency, progression
│   ├── team.py          # team KPIs, sideout breakdowns
│   └── advanced.py      # expected sideout, serve pressure, win probability
├── data/cache/          # existing, unchanged (matches.json)
├── generate_site.py     # NEW: runs analytics → exports JSON to site/data/
└── site/                # NEW: static frontend
    ├── index.html        # Overview page
    ├── players.html      # Player detail page
    ├── comparison.html   # Radar chart comparison page
    ├── css/
    │   └── style.css     # All styling, CSS variables, dark theme
    ├── js/
    │   ├── app.js        # Navigation, shared logic, data loading
    │   ├── overview.js   # Overview page charts & KPIs
    │   ├── players.js    # Player stats & drilldowns
    │   └── comparison.js # Radar charts & trends
    └── data/             # Generated by generate_site.py
        ├── overview.json
        ├── players.json
        └── comparison.json
```

## Data Pipeline

1. User runs `python generate_site.py`
2. Script loads `data/cache/matches.json`
3. Calls `analytics.core.build_all()` to get rallies and actions DataFrames
4. Computes page-specific aggregations using existing `analytics.team`, `analytics.player`, and `analytics.advanced` modules
5. Serializes results as JSON into `site/data/`
6. `site/` folder is ready to open locally or push to GitHub Pages

The existing Hudl/Balltime data pipeline (`seed_cache.py`, `data/balltime.py`) is unchanged. Users refresh data the same way as before — `seed_cache.py` or the Dash app's refresh button — then re-run `generate_site.py`.

## Pages

### Page 1: Overview (index.html)

**KPI Row (5 cards):**
- Record (W-L)
- Sideout %
- Break Point %
- Hitting Efficiency
- Pass Average

**Team Progression Chart:**
- Plotly.js line chart showing key metrics (sideout %, hitting efficiency, pass average) trending over games across the season

**Attack Stats by Game State:**
- Grouped bar chart showing team hitting efficiency broken down by score situation: Winning Big, Winning, Close, Losing, Losing Big

**Pass Stats by Game State:**
- Grouped bar chart showing team pass average broken down by the same score situations

**Game State Definitions:**
- Inline pill badges defining each state:
  - Winning Big: leading by 5+
  - Winning: leading by 2-4
  - Close: within 1 point (diff -1 to +1)
  - Losing: trailing by 2-4
  - Losing Big: trailing by 5+

**Game Results Table:**
- Sortable table with columns: Date, Opponent, Score, Result (W/L badge)

**Sideout by Game Phase:**
- Bar chart showing sideout % by Early (max score 0-9), Middle (10-19), Final (20+)

### Page 2: Players (players.html)

**Player Selector:**
- Dropdown to select a single player

**Player KPI Cards (6):**
- Kills, Hitting Efficiency, Aces, Digs, Pass Average, Errors

**Clutch Performance:**
- Side-by-side comparison bars showing clutch vs normal stats (hitting eff, kill %, pass avg)
- Clutch definition: both teams ≥20 in sets 1-2, both ≥10 in set 3

**Stats by Game State:**
- Single grouped bar chart with two series (hitting efficiency and pass average) broken down by the 5 score situations, for this player only

**Season Progression:**
- Plotly.js line chart showing the player's kills, efficiency, and pass average per match over the season

**Consistency Index:**
- Single metric badge (0-1 scale, higher = more consistent)
- Based on standard deviation of per-match hitting efficiency: score = 1 / (1 + std_dev)

### Page 3: Comparison (comparison.html)

**Multi-Player Select:**
- Select 2+ players via dropdown or clickable pills
- Each selected player gets a distinct color

**Radar Chart:**
- Plotly.js `scatterpolar` chart with overlapping polygons per selected player
- Axes: Kills, Hitting Eff, Aces, Digs, Pass Avg, Consistency
- Values normalized to 0-1 scale for comparable display

**Trend Comparison:**
- Multi-line Plotly.js chart showing selected players' key stats over the season

**Detailed Comparison Table:**
- Table with one column per selected player, one row per metric
- Best-in-class value highlighted in gold per row

## Frontend Design

The frontend-design skill will handle the creative execution. The design direction established during brainstorming:

**Theme:** Dark theme inspired by NCVA Power League Dashboard
- Dark navy background (`#0f172a` range)
- Surface cards on slightly lighter dark (`#1e293b` range)
- Slate borders (`#334155`)
- Light text (`#f1f5f9`) with muted secondary text (`#94a3b8`)
- Accent cyan (`#38bdf8`) for primary highlights

**Typography:**
- System font stack: `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif`
- Uppercase labels with letter-spacing for section headers
- Tabular numeric variants for data alignment

**Components:**
- CSS variables for consistent theming
- Cards with border-radius 8px and 1px solid borders (no heavy shadows)
- Pill badges with border-radius 999px for status indicators
- Color-coded game state pills (green for winning, red for losing, neutral for close)
- W/L badges in game results table
- Hover states and transitions (0.15s) on interactive elements

**Navigation:**
- Horizontal top nav bar with gradient header (replacing the Dash sidebar)
- Active tab indicated by accent color underline
- 3 tabs: Overview, Players, Comparison

**Charts:**
- Plotly.js via CDN
- Dark-themed chart backgrounds matching the page
- Consistent color palette across all charts

## Technology

- **Charts:** Plotly.js (CDN, same library Dash wraps — supports scatterpolar for radar charts)
- **Styling:** Single CSS file with CSS variables
- **Interactivity:** Vanilla JavaScript, per-page JS files
- **Data loading:** `fetch()` to load JSON files, filter/render client-side
- **No build tools:** No Node.js, no bundler, no framework. Plain HTML/CSS/JS.
- **Navigation:** Standard `<a>` links between HTML pages

## Deployment

**Local workflow (manual):**
1. Refresh match data if needed: `python seed_cache.py` or use existing Dash app
2. Generate static site: `python generate_site.py`
3. Push `site/` to GitHub Pages: `git subtree push --prefix site origin gh-pages` (or configure GitHub Pages to serve from `site/` on main)

**No CI/CD.** No GitHub Actions. No secrets required. Fully local generation and manual push.

## Scope — What's Included

- 3 pages: Overview, Players, Comparison
- `generate_site.py` data pipeline script
- Static frontend in `site/` folder
- GitHub Pages deployment support

## Scope — What's Excluded (deferred to future iterations)

- Runs page (scoring run analysis)
- Game Detail page (single-game momentum/win probability)
- Zones page (court heatmaps)
- Rotation-based stat breakdowns (data not available from Balltime)
- Automated deployment (GitHub Actions)
- Data refresh from the static site (use existing Dash app or seed_cache.py)

## Existing Code Reuse

The following existing modules are used as-is by `generate_site.py`:
- `analytics/core.py` — `build_all()`, `classify_score_situation()`, `classify_game_phase()`, `is_clutch()`
- `analytics/team.py` — `team_kpis()`, `sideout_by_category()`
- `analytics/player.py` — `player_season_stats()`, `clutch_comparison()`, `consistency_index()`, `season_progression()`
- `data/loader.py` — `load_data()`

No changes to existing Python code are needed. The Dash app (`app.py`, `pages/`, `components/`) continues to work independently for local interactive use.
